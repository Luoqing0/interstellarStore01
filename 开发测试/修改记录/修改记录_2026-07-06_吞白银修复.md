## 修改详情

### Bug A：遍历动态列表时 SplitOff 触发 DeSpawn，列表缩小导致跳过堆叠

**根因**: `map.listerThings.ThingsOfDef(ThingDefOf.Silver)` 返回的是 `ListerThings` 内部列表的**引用**，不是副本。`Thing.SplitOff(count)` 当 `count >= stackCount` 时会调用 `DeSpawnOrDeselect()`，从 `ListerThings` 中移除该 Thing。遍历中 `si++` 会跳过下一个堆叠。

通过 MCP 查原版源码确认：
- `ListerThings.ThingsOfDef()` → `ThingsMatching()` → 返回 `listsByDef[def]`（内部列表引用）
- `Thing.SplitOff(count)`：`count >= stackCount` 时调用 `DeSpawnOrDeselect()` + `holdingOwner.Remove(this)`

**复现场景**: 10 堆白银各 6000（总 60000），购买需 35000（约0.58倍）：
- 每扣完一堆 DeSpawn → 列表缩小 → si++ 跳过下一堆
- 实际只扣了 5 堆（30000），而非 35000
- 完美对应"购买价≥0.5倍时，白银减半"

### Bug B：扣减失败时直接 return，不退还已扣白银、不发物品

**根因**: 原代码第458-462行：
```csharp
if (剩余白银 > 0)
{
    Messages.Message("StarStore_InsufficientSilver"..., ...);
    return;  // 已扣白银未退还，物品也未发出
}
```
Bug A 导致扣减不足后，已扣的白银没有退还逻辑，`return` 跳过了物品分发。结果：白银少了，物品没到。

### Bug C：SplitOff 返回值未使用，白银未正确销毁

**根因**: `silver.SplitOff(扣除)` 返回分离出的 Thing，但返回值被丢弃。分离出的白银成为游离对象，未被 `Destroy()`。

通过 MCP 查 `Thing.SplitOff` 实现确认：
- `count < stackCount`：创建新 Thing，减少原堆叠，返回新 Thing
- `count >= stackCount`：DeSpawn + Remove，返回 this
- 两种情况返回值都需要调用者处理

## 修复方案

### 提取公共方法 `扣除白银(Map, int)`

将白银扣减逻辑提取为统一的两阶段方法，购买/捆绑包/抽卡三处共用：

**阶段 1 预检查**：遍历白银堆，收集可用堆叠到**副本列表**，累加确认总量足够。不足则返回 false，**不扣除任何白银**。

**阶段 2 扣减**：确认足够后，遍历**副本列表**统一扣减：
- `Thing split = silver.SplitOff(扣除); if (split != null && !split.Destroyed) split.Destroy();`
- 正确销毁返回值，避免游离对象

### 三处调用统一替换

| 调用位置 | 文件 | 修复内容 |
|---------|------|---------|
| 普通购买 | `MainTabWindow_星际商店_交易.cs` `执行购买()` | 替换内联扣减为 `扣除白银(map, 需要白银)` |
| 捆绑包购买 | `MainTabWindow_星际商店_捆绑包.cs` `执行购买捆绑包()` | 同上 |
| 抽卡 | `MainTabWindow_星际商店_抽卡.cs` `执行抽卡()` | 同上，并补充扣减失败检查（原代码扣减失败后直接继续抽卡） |

## 关于"显示不一致"（60000 vs 15000）

`获取白银总量` 排除了被预留（`reservationManager.IsReserved`）和被小人持有（`ThingOwnerUtility.AnyParentIs<Pawn>`）的白银。原版资源栏显示的是总白银。修复 Bug A-C 后，之前因 SplitOff 返回值未销毁导致的白银状态异常将消除，显示差异应大幅缩小。如仍有差异，属正常过滤行为（预留白银不可用于交易）。
