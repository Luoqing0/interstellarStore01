### H1. 卖出模式"全部"分类 + "仅显示库存"过滤遗漏机械族

**文件**：`MainTabWindow_星际商店_交易.cs`
**位置**：第 273-278 行（Union 添加机械族）+ 第 296-332 行（仅显示库存过滤）

**问题描述**：
在卖出模式下，当分类为 `StarStore_All`（全部）时，代码在第 273-278 行通过 `Union` 将殖民地机械族种族加入查询列表：

```csharp
// 第 273-278 行
if (!是购买模式 && 当前分类标签 == "StarStore_All")
{
    var mechRaces = 机械族管理器.获取殖民地机械族Race(Find.CurrentMap);
    if (!mechRaces.NullOrEmpty())
        query = query.Union(mechRaces);
}
```

然而在后续的"仅显示库存"过滤中（第 296-332 行），当 `当前显示机械族` 为 false 时（"全部"分类下该值为 false），走 else 分支：

```csharp
// 第 300-331 行
if (当前显示机械族)  // false for "StarStore_All"
{
    var present = 机械族管理器.获取殖民地机械族Race(map).ToHashSet();
    query = query.Where(d => present.Contains(d));
}
else
{
    HashSet<ThingDef> 拥有defs = new HashSet<ThingDef>();
    // 只检查 listerThings（物品）...
    // 只检查 mapPawns 中的动物...
    List<Pawn> allPawns = map.mapPawns.AllPawns;
    for (int i = 0; i < allPawns.Count; i++)
    {
        Pawn p = allPawns[i];
        if (p.Faction == Faction.OfPlayer && p.RaceProps.Animal && !p.Dead)  // 只检查动物！
        {
            if (!拥有defs.Contains(p.def))
                拥有defs.Add(p.def);
        }
    }
    query = query.Where(d => 拥有defs.Contains(d));  // 机械族被过滤掉
}
```

else 分支检查了 `mapPawns` 中的动物（`p.RaceProps.Animal`），但**没有检查机械族**（`p.RaceProps.IsMechanoid`）。通过 Union 添加的机械族 ThingDef 不在 `拥有defs` 中，因此被 `query.Where(d => 拥有defs.Contains(d))` 过滤掉。

**影响**：在卖出模式"全部"分类下开启"仅显示库存"时，玩家拥有的机械族不会显示在列表中。

**建议修复**：在 else 分支的 `mapPawns` 遍历中增加机械族检查：

```csharp
if (p.Faction == Faction.OfPlayer && !p.Dead &&
    (p.RaceProps.Animal || p.RaceProps.IsMechanoid))
{
    if (!拥有defs.Contains(p.def))
        拥有defs.Add(p.def);
}
```

### H2. 卖出模式 >> 按钮对有变体物品完全失效（显示 0 库存）

**文件**：`MainTabWindow_星际商店_数量控制.cs`（`绘制紧凑数量控制V2` 和 `绘制完整数量控制`）
       `MainTabWindow_星际商店_物品网格.cs`（`绘制单个物品格子`）

**位置**：
- 物品网格第 299-306 行：sell 模式下 TransactionKey 构造
- 数量控制第 107-141 行（大布局）和 第 233-265 行（中布局）：>> 按钮库存计数

**问题描述**：
在卖出模式下，`绘制单个物品格子` 中构建 TransactionKey 时，品质和材料始终为 null：

```csharp
// 物品网格.cs 第 299-305 行
QualityCategory? 当前品质 = null;
ThingDef 当前材料 = null;
if (是购买模式)  // 卖出模式不进入此分支
{
    购买品质选择.TryGetValue(def, out 当前品质);
    购买材料选择.TryGetValue(def, out 当前材料);
}
TransactionKey 价格key = new TransactionKey(def, 当前品质, 当前材料);
// 卖出模式下: 价格key = (def, null, null)
```

但在 >> 按钮的库存计数中（数量控制.cs 第 125-135 行）：

```csharp
var 库存映射数据 = 获取库存映射(map);
if (库存映射数据.TryGetValue(key.def, out List<Thing> things))
{
    for (int i = 0; i < things.Count; i++)
    {
        Thing t = things[i];
        TransactionKey tk = new TransactionKey(t);  // t 有品质/材料
        if (tk.Equals(key))  // (def, GoodQuality, Steel) != (def, null, null) -> false
            库存 += t.stackCount;
    }
}
```

`new TransactionKey(t)` 从实际物品构造，会包含品质和材料信息。而 `key` 是 `(def, null, null)`。`TransactionKey.Equals` 比较三个字段全等，因此 `tk.Equals(key)` 永远为 false——任何有 CompQuality 或 MadeFromStuff 的物品都不会被计入。

**影响**：
- >> 按钮对所有有品质/材料变体的物品显示 0 库存
- 用户直接在输入框中输入数量后点击卖出，会收到"库存不足"错误，因为卖出执行时用同样的 `(def, null, null)` key 查找候选物品，匹配不到任何有品质的库存物品

**建议修复**：
方案 A（推荐）：卖出模式下 >> 按钮统计所有变体的总数量，不按 key 精确匹配：

```csharp
// 卖出模式 >> 按钮库存计数
if (库存映射数据.TryGetValue(key.def, out List<Thing> things))
{
    for (int i = 0; i < things.Count; i++)
    {
        库存 += things[i].stackCount;  // 统计同 def 所有变体
    }
}
```

方案 B：卖出模式下对有变体物品隐藏 >> 按钮，仅保留"选择变体"入口。

### H3. 机械族/动物有 CompQuality 时变体弹窗为空

**文件**：`MainTabWindow_星际商店_物品网格.cs` 第 406-418 行
       `MainTabWindow_星际商店.cs` 第 539-552 行（`物品有变体`、`获取库存物品`）

**问题描述**：
`物品有变体` 方法检查 ThingDef 是否有 CompQuality 或 MadeFromStuff：

```csharp
public bool 物品有变体(ThingDef def)
{
    return def.HasComp(typeof(CompQuality)) || def.MadeFromStuff;
}
```

某些模组的机械族/动物 ThingDef 可能带有 CompQuality。此时在卖出模式下，`物品有变体(def)` 返回 true，显示"选择变体"按钮。但点击后：

```csharp
List<Thing> inventory = 获取库存物品(def, map);
Find.WindowStack.Add(new Dialog_SellVariant(def, inventory));
```

`获取库存物品` 调用 `获取库存映射`，而 `获取库存映射` 只收集 `ThingCategory.Item`（第 572 行 `if (thing.def.category != ThingCategory.Item) continue;`），Pawn 不在其中。因此返回空列表，`Dialog_SellVariant` 显示"暂无变体"。

**影响**：用户点击"选择变体"后看到空弹窗，无法操作。

**建议修复**：在 `物品有变体` 中排除 Pawn 类型，或对 Pawn 类型始终走 >> 按钮路径：

```csharp
public bool 物品有变体(ThingDef def)
{
    if (def.race != null) return false;  // Pawn 不走变体弹窗
    return def.HasComp(typeof(CompQuality)) || def.MadeFromStuff;
}
```

### H4. Dialog_SellVariant 数量输入框无持久化缓冲（光标跳变）

**文件**：`Dialog_SellVariant.cs` 第 179-185 行

**问题描述**：
`Dialog_SellVariant` 中每行为数量输入框每帧重新创建字符串：

```csharp
string 数量文本 = row.quantity.ToString();          // 每帧新建
数量文本 = Widgets.TextField(数量Rect, 数量文本);    // 使用局部变量
if (int.TryParse(数量文本, out int parsed))
    row.quantity = Mathf.Clamp(parsed, 0, row.stock);
```

`数量文本` 是局部变量，不跨帧持久化。用户在输入框中输入"1"后，下一帧 `row.quantity` 被解析为 1，`数量文本` 被重建为 "1"，TextField 重新渲染，光标跳到末尾。输入多位数字（如"100"）时体验极差。

主窗口的 `绘制紧凑数量控制V2` 已通过 `数量输入缓冲` 字典解决了这个问题，但 `Dialog_SellVariant` 未采用相同的方案。

**影响**：用户在变体弹窗中无法流畅输入数量。

**建议修复**：在 `VariantRow` 中增加 `string 数量缓冲` 字段，或维护一个 `Dictionary<int, string>` 缓冲（按行索引）：

```csharp
private class VariantRow
{
    public TransactionKey key;
    public int stock;
    public float unitPrice;
    public int quantity;
    public string label;
    public string 数量缓冲;  // 新增：持久化输入缓冲
}
// 构造时初始化: row.数量缓冲 = "0";
// 渲染时:
if (row.数量缓冲 == null) row.数量缓冲 = row.quantity.ToString();
数量文本 = row.数量缓冲;
数量文本 = Widgets.TextField(数量Rect, 数量文本);
row.数量缓冲 = 数量文本;
```

### M1. 执行卖出 SplitOff 返回值未销毁（资源泄漏）

**文件**：`MainTabWindow_星际商店_交易.cs` 第 649 行

**问题描述**：
卖出执行中对普通物品调用 `SplitOff` 后未处理返回值：

```csharp
t.SplitOff(本次卖出数);  // 返回值被丢弃
```

对比购买扣银代码（第 204-205 行）：
```csharp
Thing 分离白银 = 白银堆.SplitOff(扣除);
if (分离白银 != null && !分离白银.Destroyed) 分离白银.Destroy();
```

当 `本次卖出数 < t.stackCount` 时，`SplitOff` 创建一个新的未生成的 Thing 对象并返回。该对象未被引用、未被销毁、未生成到地图。虽然 C# GC 最终会回收，但 Thing 对象可能被其他系统引用追踪，存在隐患。

**建议修复**：

```csharp
Thing sold = t.SplitOff(本次卖出数);
if (sold != null && sold != t && !sold.Destroyed)
    sold.Destroy();
```

### M6. 执行卖出中机械族/动物预检查与执行之间可能不一致

**文件**：`MainTabWindow_星际商店_交易.cs` 第 544-554 行（阶段1）和 第 597-614 行（阶段2）

**问题描述**：
阶段1 预检查调用 `机械族管理器.获取殖民地机械族(kv.Key.def, map, kv.Value).Count()`，阶段2 执行再次调用 `机械族管理器.获取殖民地机械族(plan.key.def, map, plan.amount).ToList()`。两次调用之间虽然不经过游戏 tick，但 LINQ 查询每次都会重新遍历 `mapPawns.AllPawns`，且 `Take(amount)` 的数量参数不同（阶段1 用 `kv.Value`，阶段2 用 `plan.amount`，实际值相同但语义不明确）。

更重要的是，阶段2 中如果实际获取的数量少于预期（第 600-604 行的检查），会直接 return 而不回滚已处理的收益。但由于阶段1 已验证过，这种情况极少发生。

**影响**：性能浪费（重复遍历 AllPawns），极低概率的一致性问题。

**建议修复**：阶段1 收集的 Pawn 引用直接传入阶段2：

```csharp
// 阶段1
var mechs = 机械族管理器.获取殖民地机械族(kv.Key.def, map, kv.Value).ToList();
if (mechs.Count < kv.Value) { /* 不足 */ return; }
执行计划.Add(new 出售计划项 { key = kv.Key, candidates = mechs.Cast<Thing>().ToList(), amount = kv.Value });

// 阶段2 直接使用 plan.candidates，无需再次查询
```

### L4. 绘制单个物品格子每帧创建大量 Color 对象

**文件**：`MainTabWindow_星际商店_物品网格.cs` 第 128-148 行

**问题描述**：
8-strip 渐变背景每帧为每个可见格子创建 8 个 `Color` 对象（`Color.Lerp` 返回新 Color）。12 个可见格子 = 96 个 Color/帧。

**建议修复**：预计算渐变色带，缓存为静态数组：

```csharp
private static readonly Color[] 渐变色带 = 预计算色带();
private static Color[] 预计算色带()
```

### L10. 缓存格子尺寸在分页切换时不重算

**文件**：`MainTabWindow_星际商店_物品网格.cs` 第 66-79 行

**问题描述**：
格子尺寸缓存基于 `当前页物品.Count`（每页物品数量）。如果某页有 12 个物品，另一页有 8 个物品（最后一页），切换页面时 `当前页物品.Count` 变化会触发重算。但缓存条件检查 `当前页物品.Count != 上次物品数量`，如果两页物品数恰好相同，不会重算。这通常没问题（尺寸与物品数无关），但如果物品数变化导致列数不同（通过其他逻辑），可能布局异常。

**影响**：极低概率的视觉小问题。

### L12. 执行卖出中动物/机械族出售价格不含品质因素

## 五、正面评价

代码中以下方面做得较好：

1. **帧级缓存体系完善**：交易改善、白银总量、基础价格、库存映射均有帧级缓存，避免每帧重复计算
2. **两阶段白银扣减**：预检查 + 执行的模式有效避免了"吞白银"bug
3. **两阶段卖出执行**：预检查所有购物车项库存后统一执行，避免部分扣除后失败
4. **虚拟滚动优化**：物品网格只渲染可见行，大型物品列表性能良好
5. **机械族/动物 Pawn 特殊路径**：买入和卖出均有独立的 Pawn 生成/销毁逻辑，避免了 ThingMaker.MakeThing 创建畸形 Pawn 的问题
6. **库存映射优化**：用 for 循环替代 LINQ，手动过滤 thingCategories，减少 GC 压力
7. **跨天折扣刷新**：在 DoWindowContents 中检测天数变化自动刷新折扣和礼包
8. **窗口拖拽位置记忆**：通过 ModSettings 持久化窗口位置
